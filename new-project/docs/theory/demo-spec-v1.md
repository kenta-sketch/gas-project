# デモ仕様書 v1

クワッドマインド理論プロジェクト ── デモ画面開発仕様
作成: 2026-05-01
対象範囲: Option B(採用1人の入口→出口 + 同じ人物の1年後時系列)
開発環境: Claude Code(健太さん側)

---

## 1. ゴール

友人(理論作者)に「自分の理論がツールとしてこう見える」のイメージ参照点を提供する。**完成度より「絵があること」が先**。

### 成功基準
- 友人が画面を見た時、「これは自分の理論が動いている」と直感で理解できる
- 5分以内のデモで「採用→入社1年後の変化」のストーリーが見える
- 出力品質が友人提供サンプルPDF(自己分析レポート/マネジメントガイド)の水準に近い(完全一致不要、方向性の共有が目的)

### 非ゴール(デモではやらない)
- 本番機能の網羅
- 完全なセキュリティ実装
- データベース永続化
- 認証認可
- 履歴書解析、面接質問提案、整合性チェック(これらはv2以降)

※ 注: その後の方針更新で、履歴書解析は v1 に組み込み済み。整合性チェックは「整合性を見せる」ではなく「情報を確実に取る」設計に変更。

---

## 2. デモシナリオ

採用候補者「佐藤さん(20代後半・男性・営業職応募)」が診断を受け、採用判断を経て、入社後1年で再診断する5分間ストーリー。

**シーン1(採用前)**
1. 佐藤さんが診断を受ける
2. A/B/C/Dスコアと5感情スコアが算出される
3. 営業職への適合度判定が表示される
4. 自己分析レポートとマネジメントガイドが生成される

**シーン2(入社1年後)**
1. 同じ佐藤さんが再診断を受ける
2. スコアが変化している(例: A 18→22、D 20→16)
3. 「現環境での変化」コメントが自動生成される
4. 配置最適化の提案が出る

## 4. 入力仕様

### 4.1 属性データ(候補者プロフィール)
- 氏名(デモ用ダミー)
- 年代(20代前半 / 20代後半 / 30代 / 40代 / 50代以上)
- 性別
- 現職位 / 職種
- 応募職種(採用シナリオ用)

### 4.2 診断質問
- 既存Q1〜Q9を採用(マスター構造図のリンク先参照)
- 各設問に4択(A/B/C/Dそれぞれの傾向に対応)
- 別途、5感情の自己評価(各1〜5の5段階)

**注**: 現状Q1〜Q9だけではサンプルPDF出力の粒度に届かない。デモでは既存質問+選択時の重み付けで動かす。本番までに質問拡張が必要(マスター構造図のギャップ参照)。

---

## 5. スコア計算ロジック

### 5.1 A/B/C/Dスコア(各25点満点)
各質問選択肢に重み付け:
- 選択肢1 → A+一定点
- 選択肢2 → B+一定点
- 選択肢3 → C+一定点
- 選択肢4 → D+一定点

合計を25点満点に正規化。各質問の重みは初期はすべて等重で実装、後で調整可能にする。

### 5.2 5感情スコア(各5点満点)
直接5段階自己評価をそのまま採用。

### 5.3 タイプ判定ロジック
```
if D最高 かつ A < 12 → 理詰め型
elif B最高 かつ C < 12 → 承認欲求型
elif A最高 かつ B < 12 → ワガママ型
elif 全軸が15以上 かつ (最大-最小) < 8 → 統合型
else → 混合型
```

### 5.4 1年後シナリオの差分生成(デモ用簡易)
シーン1のスコアに事前定義の差分を加算する。例: A+4, B+4, D-4。実際の変化計算は v2 以降の実装。

## 6. 出力仕様

### 6.1 レーダーチャート
A/B/C/Dの4軸。Chart.js or Recharts想定。色は1色固定でOK。1年後比較画面では2スコアを重ね表示。

### 6.2 5感情バーチャート
5本の水平バー(0〜5)。並びは不安 / 悲しみ / 怒り / 喜び / 幸福。

### 6.3 自己分析レポート
- LLM API呼び出しで生成
- **プロンプトは別ファイル「llm-prompts-v1.md」参照**
- 入力: スコア / 属性 / 5感情
- 出力: サンプルPDF「自己分析レポート」と同水準のテキスト
- 表示: HTMLレンダリング(マークダウン→HTML変換でOK)

### 6.4 マネジメントガイド
- 同上、別プロンプトで生成
- 入力: 同じ
- 出力: サンプルPDF「マネジメントガイド」と同水準

### 6.5 1年後比較ビュー
- スコア差分の可視化(矢印付き or 重ね合わせレーダー)
- LLM生成の変化解釈コメント(例: 「Aが上がりB が下がっている。環境への適応が進み、感情表現が豊かになっている」)
- 配置最適化提案(例: 「現職継続が最適。1年後にチームリーダー候補として検討可能」)

## 7. 技術スタック

### 推奨(当初案)
- **フロントエンド**: HTML + Vanilla JS + Tailwind CSS(健太さんが慣れていればNext.jsでも可)
- **チャート**: Chart.js または Recharts
- **LLM API**: Claude API(Anthropic Messages API、claude-sonnet-4-6推奨)
- **デプロイ**: Vercel / Netlify(URLで友人に共有可能にする)
- **データ**: ローカルJSON + メモリ(本番ではDB必要)

### 採用結果
Next.js 15 (App Router) + TypeScript + Tailwind + Recharts + Anthropic SDK で実装。
URL保護されたAPIで L3 理論機構(プロンプト本体)を保護。Vercelにデプロイ。

## 10. 受け入れ基準

デモ完成と判定する条件:

1. ✓ 質問入力 → スコア表示 → レポート2種生成 までが1本のフローで動く
2. ✓ レポートの品質が、友人サンプルPDFを見た友人本人が「方向性は合っている」と感じるレベル
3. ✓ 1年後比較ビューで時系列変化が一目で分かる
4. ✓ 5分以内に全フローを完走できる
5. ✓ ブラウザで動く(URLで友人に共有可能)

## 11. 開発の進め方(Claude Code向けタスク分解)

エージェント実行と相性が良いように、各ステップが独立で動作確認できる粒度に分解:

1. プロジェクト初期化(リポジトリ作成、依存パッケージ、Tailwind設定)
2. 静的データ準備(/data/candidates.json、ダミー2-3名分)
3. 画面1: ホーム / 候補者選択 UI
4. 画面2: 質問入力 UI
5. スコア計算ロジック実装(セクション5の式)
6. 画面3: スコア結果(レーダーチャート + 5感情バー + タイプ判定)
7. LLM API連携基盤(API key管理、エラーハンドリング)
8. プロンプト統合①: 自己分析レポート生成
9. 画面4-A: 自己分析レポート表示
10. プロンプト統合②: マネジメントガイド生成
11. 画面4-B: マネジメントガイドタブ切替
12. 1年後シナリオ差分実装
13. 画面5: 1年後比較(差分可視化 + LLM変化解釈 + 配置提案)
14. 全体動作確認 + Vercelデプロイ
15. 友人へ共有可能なURL生成

## 13. 開発時の注意

- LLMプロンプトの品質がレポート品質を決める。プロンプト草案ができ上がってから6.3/6.4の実装に着手するのが効率的
- デモ用ダミーデータは2-3名分用意し、それぞれA優位 / D優位 / 統合型のように特徴を分ける(タイプ判定の動作確認に使える)
- 1年後シナリオの差分は、「環境ストレスでDが下がりAが回復」みたいな理論に沿った変化を選ぶ(ストーリーが立つ)
- 友人へのデモでは「採用時 vs 1年後」を並べて見せる演出を意識する(SaaS価値の核心が伝わる)
